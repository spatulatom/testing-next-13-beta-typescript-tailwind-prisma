## Migration from previous model

### Concept map

| Concept                      | Previous model (Next.js 13–15)                    | Cache Components (Next.js 16+)                                            |
| ---------------------------- | ------------------------------------------------- | ------------------------------------------------------------------------- |
| Default behavior             | Everything cached unless opted out                | Everything dynamic unless opted in                                        |
| Opt in to cache              | Route segment `export const revalidate`           | `'use cache'` directive                                                   |
| Cache a function             | `unstable_cache(fn, keys, { tags, revalidate })`  | `'use cache'` + `cacheTag()` + `cacheLife()` inside the function          |
| Cache duration               | `revalidate: 3600` (seconds)                      | `cacheLife('hours')` (named profile)                                      |
| Tag a cache entry            | Config object `{ tags: ['posts'] }`               | Function call `cacheTag('posts')` inside the function                     |
| Invalidate in Server Action  | `revalidatePath('/')` or `revalidateTag('posts')` | `updateTag('posts')` (preferred) or `revalidatePath()` still works        |
| Invalidate in Route Handler  | `revalidateTag('posts')`                          | `revalidateTag('posts', 'max')` or `revalidatePath()` still works         |
| Force dynamic                | `export const dynamic = 'force-dynamic'`          | Nothing — it's the default, just remove it                                |
| Force static                 | `export const dynamic = 'force-static'`           | `'use cache'` + `cacheLife('max')`                                        |
| Runtime APIs in cached scope | Allowed                                           | Not allowed — extract and pass as argument, or use `'use cache: private'` |
| Rendering model              | ISR / full-route static                           | Partial Prerendering (PPR)                                                |

### Code patterns

**Caching a data function**

```tsx
// Before
import { unstable_cache } from 'next/cache';

export const getPosts = unstable_cache(
  async () => db.posts.findMany(),
  ['posts'],
  { tags: ['posts'], revalidate: 3600 }
);

// After
import { cacheLife, cacheTag } from 'next/cache';

export async function getPosts() {
  'use cache';
  cacheLife('hours');
  cacheTag('posts');
  return db.posts.findMany();
}
```

**Caching a page**

```tsx
// Before
export const revalidate = 3600;

export default async function Page() {
  const posts = await getPosts();
  return <div>{posts}</div>;
}

// After
import { cacheLife } from 'next/cache';

export default async function Page() {
  'use cache';
  cacheLife('hours');
  const posts = await getPosts();
  return <div>{posts}</div>;
}
```

**Invalidating after a mutation**

```tsx
// Before — in a Server Action
import { revalidatePath } from 'next/cache';

export async function createPost(title: string) {
  await db.posts.create({ data: { title } });
  revalidatePath('/'); // blunt: nukes entire route cache
  revalidatePath('/posts'); // had to list every affected path
}

// After — in a Server Action
import { updateTag } from 'next/cache';

export async function createPost(title: string) {
  await db.posts.create({ data: { title } });
  updateTag('posts'); // surgical: only invalidates what's tagged 'posts'
}
```

## How rendering works (PPR)

Cache Components uses **Partial Prerendering (PPR)** by default. At build time Next.js renders what it can:

| Content type                              | How to write it        | Build-time behavior                                      |
| ----------------------------------------- | ---------------------- | -------------------------------------------------------- |
| Static (pure computation, module imports) | Just write it normally | Included in the static HTML shell                        |
| Cached dynamic data                       | `'use cache'`          | Fetched once, result included in shell until invalidated |
| Per-request dynamic data                  | Wrap in `<Suspense>`   | Fallback in shell, streams in at request time            |

If you have dynamic data **not** wrapped in `<Suspense>` and **not** marked `'use cache'`, Next.js throws an error at build time:

```
Error: Uncached data was accessed outside of <Suspense>
```

## Caching data or UI

Add `'use cache'` at the top of any async function or component body.

**Cache a data-fetching function:**

```tsx
import { cacheLife, cacheTag } from 'next/cache';

export async function getPosts() {
  'use cache';
  cacheLife('hours');
  cacheTag('posts');

  return db.posts.findMany();
}
```

**Cache a whole page:**

```tsx
import { cacheLife } from 'next/cache';

export default async function Page() {
  'use cache';
  cacheLife('hours');

  const posts = await getPosts();
  return (
    <ul>
      {posts.map((p) => (
        <li key={p.id}>{p.title}</li>
      ))}
    </ul>
  );
}
```

**Cache key is automatic** — function arguments and closed-over values are included automatically, so different inputs produce separate cache entries.

> Adding `'use cache'` at the **top of a file** caches all exported functions in that file.

## Cache duration — `cacheLife()`

Call `cacheLife()` inside a `'use cache'` scope to set how long the cache entry lives.

```tsx
cacheLife('seconds'); // short-lived
cacheLife('minutes');
cacheLife('hours');
cacheLife('days');
cacheLife('weeks');
cacheLife('max'); // effectively permanent until manually invalidated
```

---

## Cache tags — `cacheTag()`

Tags let you invalidate specific cached data later. Call `cacheTag()` inside any `'use cache'` scope.

```tsx
export async function getPost(id: string) {
  'use cache';
  cacheTag('posts', `post-${id}`); // multiple tags in one call

  return db.posts.findUnique({ where: { id } });
}
```

## Cache invalidation

There are two functions for invalidating by tag. They have the same name pattern but different behaviour — choose the right one for your context.

### `updateTag()` — use in Server Actions

Immediately expires the cache. The **next request waits** for fresh data (no stale content served). Use this when a user makes a change and needs to see the result straight away (read-your-own-writes).

**Only works inside Server Actions.** Throws an error anywhere else.

```tsx
'use server';
import { updateTag } from 'next/cache';

export async function createPost(formData: FormData) {
  await db.posts.create({ data: { title: formData.get('title') } });

  updateTag('posts'); // all pages showing the post list
  updateTag(`post-${id}`); // the specific post detail page
}
```

### `revalidateTag()` — use in Route Handlers / webhooks

Stale-while-revalidate: serves the cached content immediately, then fetches fresh data in the background for the next request. Use this in Route Handlers or webhook endpoints where the caller doesn't need to see the update immediately.

```tsx
// app/api/revalidate/route.ts
import { revalidateTag } from 'next/cache';

export async function POST(req: Request) {
  const { tag } = await req.json();
  revalidateTag(tag, 'max'); // 'max' = stale-while-revalidate (recommended)
  return Response.json({ revalidated: true });
}
```

**Summary:**

|              | `updateTag()`           | `revalidateTag()`                   |
| ------------ | ----------------------- | ----------------------------------- |
| Where        | Server Actions only     | Server Actions + Route Handlers     |
| Next request | Waits for fresh data    | Gets stale data, background refresh |
| Use case     | User just made a change | Webhook / background revalidation   |

### `revalidatePath()` — still available

`revalidatePath()` **did not go away** in Cache Components. It still works in both Server Actions and Route Handlers. Internally it uses auto-generated **soft tags** (e.g. `_N_T_/blog/hello`) to invalidate the route and all its ancestor layouts.

Use it when you want to invalidate by URL path rather than by tag, or when migrating incrementally and don't yet have tags in place.

```tsx
'use server';
import { revalidatePath } from 'next/cache';

export async function createPost() {
  await db.posts.create(...);
  revalidatePath('/posts'); // invalidates /posts and its layouts
}
```

**Tag-based invalidation (`updateTag` / `revalidateTag`) is preferred** for new code because it is surgical and doesn't require you to enumerate every affected path.

## Runtime data (cookies, headers, params)

`cookies()`, `headers()`, `searchParams`, and dynamic `params` are **not allowed inside `'use cache'`** — they are request-time values and would make the cache entry un-shareable.

**Pattern 1 — Extract and pass as a prop/argument (preferred):**

```tsx
// Page reads runtime data (not cached)
export default async function Page() {
  return (
    <Suspense fallback={<p>Loading...</p>}>
      <ProfileContent />
    </Suspense>
  );
}

// Intermediate component reads cookies, passes value down
async function ProfileContent() {
  const sessionId = (await cookies()).get('session')?.value;
  return <CachedProfile sessionId={sessionId} />;
}

// Cached component receives plain value — sessionId becomes part of the cache key
async function CachedProfile({ sessionId }: { sessionId: string }) {
  'use cache';
  cacheTag('users', `user-${sessionId}`);
  const data = await fetchUser(sessionId);
  return <div>{data.name}</div>;
}
```

**Pattern 2 — `'use cache: private'` (when you can't extract):**

Marks the cache entry as private (not shared across users). Allows runtime API access.

```tsx
async function UserGreeting() {
  'use cache: private';
  const theme = (await cookies()).get('theme')?.value;
  return <p>Theme: {theme}</p>;
}
```

## Non-deterministic values

`Math.random()`, `Date.now()`, `crypto.randomUUID()` produce different values on every call. Cache Components requires you to handle them explicitly.

**If you want a fresh value per request** — call `connection()` first, then wrap in `<Suspense>`:

```tsx
import { connection } from 'next/server';
import { Suspense } from 'react';

async function RequestId() {
  await connection(); // signals: this runs at request time, not build time
  const id = crypto.randomUUID();
  return <p>Request ID: {id}</p>;
}

export default function Page() {
  return (
    <Suspense fallback={null}>
      <RequestId />
    </Suspense>
  );
}
```

**If you want one stable value per cache entry** — just use `'use cache'` normally:

```tsx
export default async function Page() {
  'use cache';
  const buildId = crypto.randomUUID(); // same value for all users until invalidated
  return <p>Build: {buildId}</p>;
}
```

## Practical Pattern: Caching & Invalidation in Routes

In real apps, caching and invalidation work together across three layers:

### Layer 1: Data-fetching functions (tag at the source)

Create data functions with `'use cache'` + `cacheTag()`. This is where all caching logic lives.

```tsx
// lib/db.ts
export async function getPosts() {
  'use cache';
  cacheLife('hours');
  cacheTag('posts'); // ← Tag at the source
  return db.posts.findMany();
}

export async function getPost(id: string) {
  'use cache';
  cacheLife('hours');
  cacheTag('posts', `post-${id}`); // ← Multiple tags for surgical invalidation
  return db.posts.findUnique({ where: { id } });
}
```

### Layer 2: Routes/Pages (call cached functions)

Pages don't need `'use cache'` themselves—they just call your data functions:

```tsx
// app/posts/page.tsx
import { getPosts } from '@/lib/db';
import { AddPostForm } from '@/components/AddPostForm';

export default async function PostsPage() {
  const posts = await getPosts(); // already cached + tagged
  return (
    <div>
      <h1>Posts</h1>
      <ul>
        {posts.map((p) => (
          <li key={p.id}>{p.title}</li>
        ))}
      </ul>
      <AddPostForm />
    </div>
  );
}
```

### Layer 3: Server Actions (mutations + invalidation)

Server Actions are for **mutations only** (create, update, delete). After the mutation, call `updateTag()` to invalidate the cache for that tag:

```tsx
// app/posts/actions.ts
'use server';
import { updateTag } from 'next/cache';
import { db } from '@/lib/db';

export async function createPost(formData: FormData) {
  const title = formData.get('title');
  
  // Mutate
  await db.posts.create({ data: { title } });
  
  // Invalidate: blow away all caches tagged 'posts'
  updateTag('posts');
}

export async function deletePost(id: string) {
  await db.posts.delete({ where: { id } });
  
  // Invalidate both the list and the specific post detail
  updateTag('posts');
  updateTag(`post-${id}`);
}
```

### Full example: read → mutate → invalidate

```tsx
// app/posts/page.tsx
import { getPosts } from '@/lib/db';
import { createPost } from '@/app/posts/actions';

export default async function PostsPage() {
  const posts = await getPosts(); // ← Layer 1: cached function with tag
  
  return (
    <div>
      <ul>
        {posts.map((p) => <li key={p.id}>{p.title}</li>)}
      </ul>
      <form action={createPost}>
        <input name="title" placeholder="New post" />
        <button type="submit">Add</button>
      </form>
    </div>
  );
}

// When user submits the form:
// 1. createPost() Server Action runs
// 2. Inserts post to DB
// 3. updateTag('posts') invalidates all caches tagged 'posts'
// 4. Next page reload fetches fresh data from getPosts() → DB (not cache)
```

**Key takeaway:** Tag at the data source (Layer 1), invalidate from Server Actions (Layer 3). Pages are just glue.

## Constraints

- **Edge runtime not supported.** Cache Components requires the Node.js runtime. Remove `export const runtime = 'edge'` from any route that uses `'use cache'`. Use [Proxy routes](https://nextjs.org/docs/app/api-reference/file-conventions/proxy) if you need edge behaviour.
- **Route segment configs removed.** `export const revalidate`, `export const dynamic`, `export const fetchCache` do nothing — use `'use cache'` + `cacheLife()` instead.
- **`unstable_cache()` replaced** by `'use cache'`.
- **`revalidatePath()` still works** — it uses auto-generated soft tags internally. Tag-based invalidation (`updateTag()`/`revalidateTag()`) is preferred for new code as it is more surgical, but `revalidatePath()` remains valid and is useful for path-based invalidation or incremental migration.

- [ ] ✅ Have `cacheComponents: true` in `next.config.ts`
- [ ] Replace `unstable_cache()` → `'use cache'` directive + `cacheTag()`

- [ ] Replace `export const dynamic = 'force-dynamic'` → Remove (default now)
- [ ] Replace `export const dynamic = 'force-static'` → `'use cache'` + `cacheLife('max')`
- [ ] Prefer `updateTag('tag-name')` over `revalidatePath('/')` for new code — `revalidatePath()` still works but tag-based invalidation is more surgical
- [ ] Add `cacheTag()` calls to all data functions you want to invalidate
- [ ] Test: data should cache, mutations should invalidate via tags
