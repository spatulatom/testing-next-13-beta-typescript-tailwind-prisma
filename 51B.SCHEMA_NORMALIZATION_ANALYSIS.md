# Schema Normalization Analysis

This document walks through the `prisma/schema.prisma` of this project as a concrete
normalization exercise.

We start from a completely flat, denormalized table and then normalize it step by step:
**0NF → 1NF → 2NF → 3NF**.

Each step is shown twice:

1. In **Prisma model syntax** (what we actually use)
2. In **SQL syntax** (`CREATE TABLE` statements)

## 1. The Starting Point: 0NF (Everything Flat)

Imagine you know nothing about database design.
You just dump every piece of data you want to track into **one giant table**.

```
user_activity (the "god table")

user_id | user_name | user_email  | post_id | post_title  | post_published | comment_id | comment_title | heart_id | account_provider | session_token
--------|-----------|-------------|---------|-------------|----------------|------------|---------------|----------|-----------------|---------------
u1      | Alice     | a@email.com | p1      | Hello World | true           | c1         | Great post!   | h1       | google           | tok_abc
u1      | Alice     | a@email.com | p1      | Hello World | true           | c2         | Love this     | h1       | google           | tok_abc
u1      | Alice     | a@email.com | p2      | Next.js Tips| false          | NULL       | NULL          | NULL     | google           | tok_abc
u2      | Bob       | b@email.com | p3      | My First    | true           | c3         | Nice one      | h2       | github           | tok_xyz
```

### What Is Wrong Here?

**Duplication everywhere:**

- Alice appears on three rows just because she has two posts and one post has two comments
- `a@email.com` is copied on every single row that belongs to Alice
- `p1 / Hello World / true` is repeated because post p1 has two comments

**Update anomaly:**

- If Alice changes her email you must update every row where she appears
- Miss one row → the database has two different emails for the same person → corrupt data

**Insert anomaly:**

- You cannot add a new user until they have at least one post (because post_id is part of the row)

**Delete anomaly:**

- If Bob deletes his only post, you also lose his account and session information because it is all in the same row

**Repeating groups:**

- A user has multiple posts. A post has multiple comments. A user has multiple hearts.
- These "many" relationships cannot be stored cleanly in one row — they explode the row count.

This table is called **0NF** (not normalized at all).

## 2. Step 1 – First Normal Form (1NF)

### The Rule

> Every column holds exactly **one atomic value**.  
> No repeating groups. No lists inside a cell.  
> Every row is uniquely identifiable.

### What We Change

The only way to keep everything in one table and satisfy "one value per cell" is to make
every row represent one unique **combination** of entities.

The composite primary key becomes: **(user_id, post_id, comment_id)**

```
user_post_comment_flat (1NF — one big table, but atomic)

PRIMARY KEY = (user_id, post_id, comment_id)

user_id | post_id | comment_id | user_name | user_email  | post_title   | post_published | comment_title
--------|---------|------------|-----------|-------------|--------------|----------------|---------------
u1      | p1      | c1         | Alice     | a@email.com | Hello World  | true           | Great post!
u1      | p1      | c2         | Alice     | a@email.com | Hello World  | true           | Love this
u1      | p2      | NULL       | Alice     | a@email.com | Next.js Tips | false          | NULL
u2      | p3      | c3         | Bob       | b@email.com | My First     | true           | Nice one
```

### Why This Is 1NF Now

- Each cell holds one value (no comma-separated lists)
- Each row is uniquely identified by the three-column composite key
- No repeating groups inside a single cell

### What Is Still Wrong (Why We Need 2NF)

Even though we fixed the "one value per cell" problem, the data is still redundant.
The reason is **partial dependencies** — a non-key column depends on only _part_ of
the composite key, not the whole thing.

| Column           | Depends on...     | Problem?                 |
| ---------------- | ----------------- | ------------------------ |
| `user_name`      | `user_id` only    | Partial dependency — bad |
| `user_email`     | `user_id` only    | Partial dependency — bad |
| `post_title`     | `post_id` only    | Partial dependency — bad |
| `post_published` | `post_id` only    | Partial dependency — bad |
| `comment_title`  | `comment_id` only | Partial dependency — bad |

Alice's name depends only on her user_id.
You do NOT need to know the post_id or comment_id to know her name.
But the composite key is `(user_id, post_id, comment_id)` — so `user_name` only partially
depends on that key. That is a violation of 2NF.

### Role of Keys at This Stage

The composite primary key `(user_id, post_id, comment_id)` guarantees uniqueness of rows
but exposes partial dependencies. The key is doing its job — it is the data grouping
around it that is wrong.

## 3. Step 2 – Second Normal Form (2NF)

### The Rule

> Already 1NF, **plus**:  
> Every non-key column must depend on the **whole** primary key — not just part of it.

### What We Change

We break apart every group of columns that "belongs together" into its own table.
Each table gets its own simple primary key (`id`).
Tables are connected using **foreign keys** — a column in one table that points to the
primary key of another table.

```
users
PK → id
-------------------------------
id | name  | email       | emailVerified | image
u1 | Alice | a@email.com | NULL          | NULL
u2 | Bob   | b@email.com | NULL          | NULL


posts
PK → id       FK → userId references users.id
--------------------------------------------------
id | title        | published | createdAt  | userId
p1 | Hello World  | true      | 2025-01-01 | u1      ← points back to Alice
p2 | Next.js Tips | false     | 2025-01-02 | u1      ← also Alice
p3 | My First     | true      | 2025-01-03 | u2      ← points to Bob


comments
PK → id       FK → postId references posts.id    FK → userId references users.id
----------------------------------------------------------------------------------
id | title       | createdAt  | postId | userId
c1 | Great post! | 2025-01-04 | p1     | u2     ← Bob commented on Alice's p1
c2 | Love this   | 2025-01-05 | p1     | u2     ← Bob commented again
c3 | Nice one    | 2025-01-06 | p3     | u1     ← Alice commented on Bob's p3


hearts
PK → id       FK → postId references posts.id    FK → userId references users.id
UNIQUE constraint on (postId, userId) — one user can heart a post only once
----------------------------------------------------------------------------------
id | postId | userId
h1 | p1     | u2     ← Bob hearted Alice's post
h2 | p3     | u1     ← Alice hearted Bob's post
```

### What Changed and Why

**Before (1NF):** `user_name` was repeated on every row involving that user.  
**After (2NF):** `name` lives in `users` exactly once. Posts, comments, and hearts all
_refer_ to the user via `userId` — the foreign key.

Changing Alice's email? Update **one row** in `users`. All posts, comments, and hearts
automatically reflect the change because they store only `userId`, not the email itself.

### How Foreign Keys Work Here

A **foreign key** is a column that holds the value of another table's primary key.
It creates a _link_ between two tables.

```
posts.userId = "u1"
                ↕  this link is the foreign key relationship
users.id    = "u1"
```

The database enforces this link. You cannot create a post with `userId = "u99"` if there
is no user with `id = "u99"`. This is called **referential integrity**.

### The Unique Composite Constraint on Heart

`Heart` has no natural single-column primary key that encodes business meaning.
But there is a business rule: **one user can heart one post only once**.

This is enforced with a **unique constraint on two columns**: `(postId, userId)`.
This is similar to a composite primary key — it prevents duplicate hearts.

In Prisma:

```prisma
@@unique([postId, userId])
```

In SQL:

```sql
UNIQUE (postId, userId)
```

### Why This Is 2NF Now

Every non-key column now depends on the **whole** primary key of its own table:

| Table    | PK  | Non-key columns depend on... | Partial dep? |
| -------- | --- | ---------------------------- | ------------ |
| users    | id  | id alone                     | No ✓         |
| posts    | id  | id alone                     | No ✓         |
| comments | id  | id alone                     | No ✓         |
| hearts   | id  | id alone                     | No ✓         |

`userId` in `posts` is a **foreign key** — it is not a "fact about the post", it is a
pointer. Foreign keys are allowed to depend on something outside the table. That is
their whole purpose.

### What Is Still Left to Check (Why We Need 3NF)

Even in 2NF, a hidden redundancy can exist: a **transitive dependency**.
This is when a non-key column depends on the key, but only _through_ another non-key column.

Example of a transitive dependency (hypothetical, not in this schema):

```
posts_with_author_country
PK → id

id | title       | userId | user_country | user_country_name
p1 | Hello World | u1     | US           | United States
```

Here:

- `user_country` depends on `userId` (which depends on `id`) → indirect
- `user_country_name` depends on `user_country` → it does NOT directly depend on `id`

Chain: `id → userId → user_country → user_country_name`

That chain — `id → non-key → non-key` — is a transitive dependency and violates 3NF.

## 4. Step 3 – Third Normal Form (3NF)

### The Rule

> Already 2NF, **plus**:  
> Every non-key column must depend **directly** on the primary key —  
> not through another non-key column.

### Checking Each Table in This Schema

**users**

```
id → name           ✓ direct
id → email          ✓ direct
id → emailVerified  ✓ direct
id → image          ✓ direct
```

No transitive dependencies. ✓

**posts**

```
id → title      ✓ direct
id → published  ✓ direct
id → createdAt  ✓ direct
id → updatedAt  ✓ direct
id → userId     ✓ direct (userId is a FK, that is fine)
```

No transitive dependencies. ✓

**comments**

```
id → title     ✓ direct
id → createdAt ✓ direct
id → postId    ✓ direct (FK)
id → userId    ✓ direct (FK)
```

No transitive dependencies. ✓

**hearts**

```
id → postId ✓ direct (FK)
id → userId ✓ direct (FK)
```

No transitive dependencies. ✓

**accounts**

```
id → userId            ✓ direct (FK)
id → type              ✓ direct
id → provider          ✓ direct
id → providerAccountId ✓ direct
id → refresh_token     ✓ direct
id → access_token      ✓ direct
id → expires_at        ✓ direct
id → token_type        ✓ direct
id → scope             ✓ direct
id → id_token          ✓ direct
id → session_state     ✓ direct
```

No transitive dependencies. ✓

**sessions**

```
id → sessionToken ✓ direct
id → userId       ✓ direct (FK)
id → expires      ✓ direct
```

No transitive dependencies. ✓

**verificationTokens**

```
(identifier, token) is the composite PK
identifier + token → expires ✓ direct
```

No transitive dependencies. ✓

### Conclusion

**The schema is already in 3NF.** No changes were needed at this step.

The final schema in Prisma (which is exactly `prisma/schema.prisma`) is:

```prisma
model User {
  id            String    @id @default(cuid())
  name          String?
  email         String?   @unique
  emailVerified DateTime?
  image         String?
  accounts      Account[]   // virtual — means Account has FK → User
  sessions      Session[]   // virtual — means Session has FK → User
  posts         Post[]      // virtual — means Post has FK → User
  comments      Comment[]   // virtual — means Comment has FK → User
  hearts        Heart[]     // virtual — means Heart has FK → User
}

model Post {
  id        String    @id @default(cuid())
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  title     String
  published Boolean   @default(false)
  userId    String               // ← foreign key
  user      User      @relation(fields: [userId], references: [id])
  comments  Comment[]
  hearts    Heart[]
}

model Heart {
  id     String @id @default(cuid())
  postId String               // ← foreign key
  userId String               // ← foreign key
  post   Post   @relation(fields: [postId], references: [id], onDelete: Cascade)
  user   User   @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@unique([postId, userId])  // ← one heart per user per post
}

model Comment {
  id        String   @id @default(cuid())
  title     String
  createdAt DateTime @default(now())
  postId    String               // ← foreign key
  userId    String               // ← foreign key
  post      Post     @relation(fields: [postId], references: [id], onDelete: Cascade)
  user      User     @relation(fields: [userId], references: [id])
}

model Account {
  id                String  @id @default(cuid())
  userId            String               // ← foreign key
  type              String
  provider          String
  providerAccountId String
  refresh_token     String? @db.Text
  access_token      String? @db.Text
  expires_at        Int?
  token_type        String?
  scope             String?
  id_token          String? @db.Text
  session_state     String?
  user              User    @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@unique([provider, providerAccountId])
}

model Session {
  id           String   @id @default(cuid())
  sessionToken String   @unique
  userId       String               // ← foreign key
  expires      DateTime
  user         User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}

model VerificationToken {
  identifier String
  token      String   @unique
  expires    DateTime
  @@unique([identifier, token])   // ← composite PK in disguise
}
```

**Key observation about Prisma's virtual fields:**  
Lines like `accounts Account[]` or `posts Post[]` in the User model are **not real database
columns**. They are Prisma's way of saying "I know there is a foreign key on the other
side that points here — let me give you a convenient way to query through it."  
The actual foreign key (`userId`, `postId`, etc.) always lives on the **many side** of
the relationship (the table with potentially many rows per parent).

## 5. The Same Journey in SQL

SQL `CREATE TABLE` statements at each normalization stage.

---

### 0NF — The God Table

```sql
-- Everything crammed into one table. Do not do this.
CREATE TABLE user_activity (
  user_id          TEXT,
  user_name        TEXT,
  user_email       TEXT,
  post_id          TEXT,
  post_title       TEXT,
  post_published   BOOLEAN,
  comment_id       TEXT,
  comment_title    TEXT,
  heart_id         TEXT,
  account_provider TEXT,
  session_token    TEXT
  -- No real primary key possible — rows are not uniquely identifiable
);
```

Problems: duplication, update/insert/delete anomalies as described in section 1.

---

### 1NF — One Flat Table, Atomic Values, Composite PK

```sql
-- We flatten to one row per (user, post, comment) combination.
-- Now every cell holds one value, but partial dependencies remain.

CREATE TABLE user_post_comment_flat (
  user_id         TEXT     NOT NULL,
  post_id         TEXT     NOT NULL,
  comment_id      TEXT     NOT NULL,   -- NULL if post has no comments yet
  user_name       TEXT,
  user_email      TEXT,
  post_title      TEXT,
  post_published  BOOLEAN,
  comment_title   TEXT,
  comment_created TIMESTAMPTZ,

  PRIMARY KEY (user_id, post_id, comment_id)
  -- user_name only depends on user_id       ← partial dependency (1NF allows, 2NF fixes)
  -- post_title only depends on post_id      ← partial dependency
  -- comment_title only depends on comment_id ← partial dependency
);
```

---

### 2NF — Separate Tables, Foreign Keys, No Partial Dependencies

```sql
-- Each entity gets its own table.
-- Non-key columns depend on the whole PK of their own table.
-- Tables are connected with FOREIGN KEY constraints.

CREATE TABLE users (
  id             TEXT        PRIMARY KEY,
  name           TEXT,
  email          TEXT        UNIQUE,
  email_verified TIMESTAMPTZ,
  image          TEXT
);

-- posts.user_id is a FOREIGN KEY → points to users.id
CREATE TABLE posts (
  id          TEXT        PRIMARY KEY,
  title       TEXT        NOT NULL,
  published   BOOLEAN     NOT NULL DEFAULT false,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL,
  user_id     TEXT        NOT NULL,

  FOREIGN KEY (user_id) REFERENCES users (id)
  -- Every post must belong to a real user.
  -- If you try to insert a post with user_id = 'ghost', the DB rejects it.
);

-- comments.post_id → posts.id   and   comments.user_id → users.id
CREATE TABLE comments (
  id         TEXT        PRIMARY KEY,
  title      TEXT        NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  post_id    TEXT        NOT NULL,
  user_id    TEXT        NOT NULL,

  FOREIGN KEY (post_id) REFERENCES posts (id) ON DELETE CASCADE,
  -- ON DELETE CASCADE: if the post is deleted, its comments are deleted too
  FOREIGN KEY (user_id) REFERENCES users (id)
);

-- hearts.post_id → posts.id   and   hearts.user_id → users.id
-- UNIQUE(post_id, user_id) prevents one user from hearting the same post twice
CREATE TABLE hearts (
  id      TEXT PRIMARY KEY,
  post_id TEXT NOT NULL,
  user_id TEXT NOT NULL,

  FOREIGN KEY (post_id) REFERENCES posts (id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users (id) ON DELETE CASCADE,

  UNIQUE (post_id, user_id)
  -- This is the SQL equivalent of Prisma's @@unique([postId, userId])
);

-- OAuth provider accounts linked to a user
CREATE TABLE accounts (
  id                  TEXT PRIMARY KEY,
  user_id             TEXT    NOT NULL,
  type                TEXT    NOT NULL,
  provider            TEXT    NOT NULL,
  provider_account_id TEXT    NOT NULL,
  refresh_token       TEXT,
  access_token        TEXT,
  expires_at          INTEGER,
  token_type          TEXT,
  scope               TEXT,
  id_token            TEXT,
  session_state       TEXT,

  FOREIGN KEY (user_id) REFERENCES users (id) ON DELETE CASCADE,

  UNIQUE (provider, provider_account_id)
  -- One account per provider per user (e.g. only one Google account per user)
);

-- Active login sessions
CREATE TABLE sessions (
  id            TEXT        PRIMARY KEY,
  session_token TEXT        UNIQUE NOT NULL,
  user_id       TEXT        NOT NULL,
  expires       TIMESTAMPTZ NOT NULL,

  FOREIGN KEY (user_id) REFERENCES users (id) ON DELETE CASCADE
);

-- Email verification tokens (no surrogate PK — the token itself is the identity)
CREATE TABLE verification_tokens (
  identifier TEXT        NOT NULL,
  token      TEXT        UNIQUE NOT NULL,
  expires    TIMESTAMPTZ NOT NULL,

  UNIQUE (identifier, token)
  -- Composite unique constraint = composite PK in disguise
  -- Prisma's @@unique([identifier, token])
);
```

---

### 3NF — No Transitive Dependencies (Already Achieved Above)

The 2NF tables above have no transitive dependencies, so the 3NF SQL is **identical**
to the 2NF SQL. No changes needed.

To illustrate what a transitive dependency would look like (and how to fix it), here
is a hypothetical example:

```sql
-- HYPOTHETICAL — this table has a transitive dependency
-- posts_with_region contains: id → user_id → region_code → region_name
-- region_name does not depend on id directly; it depends on region_code

CREATE TABLE posts_with_region (
  id          TEXT PRIMARY KEY,
  title       TEXT,
  user_id     TEXT,
  region_code TEXT,
  region_name TEXT   -- ← depends on region_code, not on id directly
  -- This violates 3NF: id → user_id → region_code → region_name
);

-- The fix: pull the transitive data into its own table
CREATE TABLE regions (
  code TEXT PRIMARY KEY,
  name TEXT NOT NULL
);

CREATE TABLE posts_fixed (
  id          TEXT PRIMARY KEY,
  title       TEXT,
  user_id     TEXT,
  region_code TEXT,

  FOREIGN KEY (region_code) REFERENCES regions (code)
  -- Now region_name lives in regions, not here. No transitive dependency.
);
```

This hypothetical violation does not exist in the actual schema of this project.
Every non-key column depends directly on its table's primary key.

## 6. Quick Reference Summary

### The Journey at a Glance

| Stage | What Was Fixed                     | Tool Used                   |
| ----- | ---------------------------------- | --------------------------- |
| 0NF   | Nothing — everything in one row    | —                           |
| 1NF   | Atomic values, no repeating groups | Composite primary key       |
| 2NF   | Partial dependencies removed       | Split tables + foreign keys |
| 3NF   | Transitive dependencies removed    | (already clean here)        |

### Key Concepts Recap

| Concept                   | What It Means in Plain English                                                      |
| ------------------------- | ----------------------------------------------------------------------------------- |
| **Primary key**           | The column(s) that uniquely identify one row in a table                             |
| **Foreign key**           | A column that holds the primary key value of a row in _another_ table               |
| **Referential integrity** | The DB enforces that a FK value must point to a row that actually exists            |
| **Partial dependency**    | A non-key column depends on only _part_ of a composite key                          |
| **Transitive dependency** | A non-key column depends on the key only _through_ another non-key column           |
| **ON DELETE CASCADE**     | When a parent row is deleted, all child rows pointing to it are also deleted        |
| **UNIQUE constraint**     | Prevents two rows from having the same value(s) in the specified column(s)          |
| **Composite unique**      | A unique constraint spanning two or more columns (e.g. one heart per user per post) |

### The Two Things Foreign Keys Do

1. **Enforce validity** — you cannot insert a post with a `userId` that does not exist in `users`
2. **Enable joins** — when you query, you can "follow" the FK link to get data from the related table

```sql
-- Example: get all posts with their author's name
SELECT posts.title, users.name
FROM posts
JOIN users ON posts.user_id = users.id;

-- The FK posts.user_id → users.id is what makes this JOIN possible and meaningful.
```

### Prisma's Virtual Relation Fields vs Real FK Columns

```
In posts table:
  userId   TEXT   ← this is a REAL column, stored in the database, it is the FK
  user     User   ← this is VIRTUAL, Prisma only, does not exist in the DB

In users table:
  posts    Post[] ← this is VIRTUAL, Prisma only, does not exist in the DB
                    it just says "I know posts has a FK pointing here"
```

The real FK always lives on the **many side** (posts, comments, hearts) — the side
that can have multiple rows per parent. The parent side (users) only gets a virtual
backlink in Prisma.
