# Normalization Examples — Step by Step

A growing collection of real tables analyzed for 1NF, 2NF, and 3NF.

---

## Example 1: `pracownicy` (Employees)

### The Original Table

```
pracownicy

Id | Imię   | Nazwisko | PESEL       | Pensja
---|--------|----------|-------------|-------
1  | Jan    | Topa     | 68010333546 | 1400
2  | Monika | Stachura | 78022212121 | 3400
3  | Michał | Posek    | 87010234567 | 5400
4  | Jan    | Mara     | 84081222000 | 2000
5  | Anna   | Rożek    | 76012900128 | 2500
```

Column translations:

- `Id` — surrogate primary key
- `Imię` — first name
- `Nazwisko` — last name
- `PESEL` — Polish national ID (unique per person, like SSN)
- `Pensja` — salary

---

### Is it already 1NF?

**Yes. Here's the checklist:**

| 1NF Rule                           | Met? | Why                                     |
| ---------------------------------- | ---- | --------------------------------------- |
| Every cell holds one atomic value  | ✓    | No lists, no comma-separated values     |
| Every row is uniquely identifiable | ✓    | `Id` is the primary key                 |
| No repeating groups                | ✓    | No column like `phone1, phone2, phone3` |

**Conclusion: already in 1NF.**

---

### Is it already 2NF?

**Yes — automatically. Here's why:**

2NF only becomes relevant when the primary key is **composite** (made of 2+ columns).

A **partial dependency** means: a non-key column depends on _only part_ of a composite key.

Here the primary key is just `Id` — a single column. You cannot have a "partial" dependency on a single column. There is no "part" to partially depend on.

**So any table with a single-column primary key is automatically in 2NF.**

Dependency check:

| Column   | Depends on `Id`? | Partial dependency?                |
| -------- | ---------------- | ---------------------------------- |
| Imię     | ✓ yes            | Not possible — PK is single column |
| Nazwisko | ✓ yes            | Not possible — PK is single column |
| PESEL    | ✓ yes            | Not possible — PK is single column |
| Pensja   | ✓ yes            | Not possible — PK is single column |

**Conclusion: already in 2NF.**

---

### Is it already 3NF?

**Yes. Here's the check:**

3NF asks: is there a **transitive dependency**?

> Transitive dependency = `Id → non-key column A → non-key column B`
> (B depends on A, not directly on Id)

Check each column against every other non-key column:

| Does X determine Y?                 | X                                                                                                                                                                        | Y   | Transitive chain? |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --- | ----------------- |
| Does `Imię` determine `Pensja`?     | No — two Jans (Id 1 and 4) have different salaries                                                                                                                       |     | No ✓              |
| Does `Nazwisko` determine `Pensja`? | No — different people, different salaries                                                                                                                                |     | No ✓              |
| Does `PESEL` determine `Pensja`?    | PESEL uniquely identifies a person, and that person has one salary — but so does `Id`. This is not a chain through a _non-key_, it's two keys pointing to the same fact. |     | No ✓              |

**No transitive dependencies found.**

**Conclusion: already in 3NF.**

---

### Side note: PESEL is a candidate key

`PESEL` is unique per row — no two employees share a national ID.
That makes it a **candidate key** — it _could_ have been chosen as the primary key instead of `Id`.

This is fine. Having two unique identifiers does not break any normal form.

---

### What would a 3NF violation look like here?

The table as-is is clean. But if we added a `department_id` and `department_name` column:

```
Id | Imię   | Pensja | department_id | department_name
---|--------|--------|---------------|----------------
1  | Jan    | 1400   | D1            | Engineering
2  | Monika | 3400   | D1            | Engineering
3  | Michał | 5400   | D2            | Marketing
```

Now we have a transitive chain:

```
Id → department_id → department_name
```

`department_name` does not depend directly on `Id`.
It depends on `department_id`, which depends on `Id`.

**That violates 3NF.** The fix:

```
pracownicy (employees)
Id | Imię   | Pensja | department_id
---|--------|--------|---------------
1  | Jan    | 1400   | D1
2  | Monika | 3400   | D1
3  | Michał | 5400   | D2

departments
department_id | department_name
--------------|----------------
D1            | Engineering
D2            | Marketing
```

Now `department_name` lives in `departments`, where it belongs.
`pracownicy` stores only `department_id` — a **foreign key** pointing to `departments`.

---

### Summary

| Normal Form | This table | Reason                                        |
| ----------- | ---------- | --------------------------------------------- |
| 1NF         | ✓ Already  | Atomic values, single PK, no repeating groups |
| 2NF         | ✓ Already  | Single-column PK — partial deps impossible    |
| 3NF         | ✓ Already  | No transitive dependencies                    |

The table is already fully normalized. The interesting teaching moment is the **hypothetical violation** added in the 3NF section above.
