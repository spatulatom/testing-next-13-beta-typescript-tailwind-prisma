## 1. Core Database Terms

### Big Picture

Think of a database like a digital filing cabinet:

- database = the whole filing system
- table = one folder or sheet
- row = one item inside it
- column = one kind of detail about that item

### Quick Visual

```text
DATABASE
|
+-- students table
|   |
|   +-- row 1 -> one student
|   +-- row 2 -> one student
|   +-- columns -> id, name, email
|
+-- courses table
|
+-- teachers table
```

### What Is a Database?

A database is an organized collection of information.

Instead of keeping information on paper, a database stores it in a structured way so it can be searched, updated, and reused.

Example:
A school database might store information about students, teachers, classes, and grades.

### What Is a Table?

A table stores one kind of thing.

Examples:

- `students` stores student information
- `courses` stores course information
- `employees` stores employee information

You can think of a table like one spreadsheet tab.

### What Is a Row?

A row is one single record in a table.

Each row represents one specific item.

Example in a `students` table:

- one row might represent Alice
- one row might represent Bob

Short version:

- table = collection of records
- row = one record

### What Is a Column?

A column is one category of information in a table.

Each column describes one property of the thing being stored.

Example columns in a `students` table:

- `student_id`
- `name`
- `email`
- `date_of_birth`

Short version:

- row = one student
- column = one type of student information

### Visual Table Example

Here is a small `students` table:

| student_id | name  | email           |
| ---------- | ----- | --------------- |
| 1          | Alice | alice@email.com |
| 2          | Bob   | bob@email.com   |

How to read it:

- the table is `students`
- the columns are `student_id`, `name`, and `email`
- each row is one student
- each individual value is a field

### Labeled Version

```text
Table name: students

                 columns
      +-------------+-------+-----------------+
      | student_id  | name  | email           |
+-----+-------------+-------+-----------------+
| row | 1           | Alice | alice@email.com |
+-----+-------------+-------+-----------------+
| row | 2           | Bob   | bob@email.com   |
+-----+-------------+-------+-----------------+

Field example: "Alice"
```

### What Is an Attribute?

An attribute is a property or characteristic of something.

In databases, an attribute usually becomes a column.

Example:
If the entity is `Student`, its attributes might be:

- `student_id`
- `name`
- `email`
- `age`

Those attributes would usually become columns in the `students` table.

### What Is a Record?

A record is another word for a row.

These two words are often used interchangeably.

Example record:

| student_id | name  | email           |
| ---------- | ----- | --------------- |
| 1          | Alice | alice@email.com |

That one line is one record, or one row.

### What Is a Field?

A field is one individual value at the intersection of a row and a column.

Example:

| student_id | name  | email           |
| ---------- | ----- | --------------- |
| 1          | Alice | alice@email.com |

In this row:

- `1` is a field in the `student_id` column
- `Alice` is a field in the `name` column
- `alice@email.com` is a field in the `email` column

### What Is an Entity?

An entity is the real-world thing you want to store information about.

Examples:

- Student
- Employee
- Product
- Order

Usually:

- entity becomes a table
- attributes become columns
- each specific instance becomes a row

Example:

```text
Entity: Student
Table: students
Attributes: student_id, name, email
One row: Alice
```

### One Simple Mapping

This is the core idea:

```text
Real world                  Database
-----------                 --------
Student                     students table
Student name                name column
One specific student        one row
"Alice"                    one field value
```

### Basic Terms Summary

| Term      | Simple meaning                            |
| --------- | ----------------------------------------- |
| Database  | A structured collection of data           |
| Table     | A collection of related rows              |
| Row       | One record in a table                     |
| Column    | One category of data in a table           |
| Attribute | A property of an entity                   |
| Record    | Another word for a row                    |
| Field     | One single value in a row/column position |
| Entity    | The real-world object being described     |

### Why These Basics Matter

These terms are the foundation for understanding:

- primary keys
- foreign keys
- relationships between tables
- normalization
- SQL queries

If these basics are clear, concepts like 1NF, 2NF, and 3NF become much easier to understand.

## 2. Keys and Relationships

### Why Keys Matter

Keys help a database do two jobs:

- uniquely identify a row
- connect one table to another table

Without keys, data becomes hard to find, update, and relate correctly.

### Primary Key

A primary key is the column, or set of columns, that uniquely identifies each row in a table.

Example:

| student_id | name  | email           |
| ---------- | ----- | --------------- |
| 1          | Alice | alice@email.com |
| 2          | Bob   | bob@email.com   |

Here, `student_id` is the primary key because each student has a different ID.

Quick rule:

- primary key = unique row identifier

### Foreign Key

A foreign key is a column that points to the primary key of another table.

Example:

`students`

| student_id | name  |
| ---------- | ----- |
| 1          | Alice |
| 2          | Bob   |

`enrollments`

| enrollment_id | student_id | course_name |
| ------------- | ---------- | ----------- |
| 10            | 1          | Math        |
| 11            | 2          | Biology     |

In `enrollments`, `student_id` is a foreign key because it points to `students.student_id`.

Quick rule:

- primary key = identifies this table's row
- foreign key = points to a row in another table

### Composite Key

A composite key is a primary key made from more than one column.

Example:

| student_id | course_id | grade |
| ---------- | --------- | ----- |
| 1          | 101       | A     |
| 1          | 102       | B     |
| 2          | 101       | A     |

If one student can take many courses, and one course can have many students, then the combination of `student_id` and `course_id` can identify each row.

So:

- `student_id` alone is not enough
- `course_id` alone is not enough
- `student_id + course_id` together are enough

### Relationship Basics

Relationships describe how tables are connected.

The three common relationship types are:

- one-to-one
- one-to-many
- many-to-many

### One-to-One

One row in table A matches one row in table B.

Example:

- one user has one profile
- one profile belongs to one user

```text
users           profiles
-----           --------
1 user   <->    1 profile
```

### One-to-Many

One row in table A can match many rows in table B.

Example:

- one customer can place many orders
- each order belongs to one customer

```text
customers       orders
---------       ------
1 customer  ->  many orders
```

### Many-to-Many

Many rows in table A can match many rows in table B.

Example:

- many students can take many courses

This is usually handled with a third table called a junction table or join table.

```text
students    ->    enrollments    <-    courses
```

### Small Relationship Example

```text
students
- student_id (PK)
- name

courses
- course_id (PK)
- title

enrollments
- student_id (FK)
- course_id (FK)
```

This means:

- `students` and `courses` have a many-to-many relationship
- `enrollments` is the bridge between them

### Keys and Relationships Summary

| Term          | Simple meaning                   |
| ------------- | -------------------------------- |
| Primary key   | Uniquely identifies a row        |
| Foreign key   | Refers to a row in another table |
| Composite key | A key made from multiple columns |
| One-to-one    | One row matches one row          |
| One-to-many   | One row matches many rows        |
| Many-to-many  | Many rows match many rows        |

## 3. Normalization from Scratch

### What Normalization Means

Normalization is the process of organizing data into tables so the data is cleaner, less repetitive, and easier to maintain.

The goals are:

- reduce duplicate data
- prevent update mistakes
- make relationships clearer

### Why Normalization Exists

If you keep too much data in one table, problems appear.

Example bad table:

| student_id | student_name | course_name | teacher_name |
| ---------- | ------------ | ----------- | ------------ |
| 1          | Alice        | Math        | Mr. Lee      |
| 1          | Alice        | Biology     | Ms. Chen     |
| 2          | Bob          | Math        | Mr. Lee      |

Problems:

- `Alice` is repeated
- `Math` is repeated
- `Mr. Lee` is repeated
- changing one fact may require updating many rows

### The Basic Idea

Normalization usually means:

1. separate different kinds of data into different tables
2. store each fact once when possible
3. connect the tables with keys

### Step 1: Start with One Big Table

```text
student_course_info
- student_id
- student_name
- course_id
- course_name
- teacher_name
```

This mixes student facts and course facts in one place.

### Step 2: Split by Entity

Instead of one big table, separate the entities:

```text
students
- student_id
- student_name

courses
- course_id
- course_name
- teacher_name

enrollments
- student_id
- course_id
```

Now:

- student information lives in `students`
- course information lives in `courses`
- who takes what lives in `enrollments`

### 1NF in Plain Language

First Normal Form means:

- each cell should hold one value
- each row should represent one thing
- there should not be lists inside a single cell

Bad:

| student_id | name  | courses       |
| ---------- | ----- | ------------- |
| 1          | Alice | Math, Biology |

Better:

| student_id | name  | course  |
| ---------- | ----- | ------- |
| 1          | Alice | Math    |
| 1          | Alice | Biology |

### 2NF in Plain Language

Second Normal Form means:

- the table is already in 1NF
- if the key has multiple columns, non-key data must depend on the whole key

Example:

| student_id | course_id | student_name | course_name |
| ---------- | --------- | ------------ | ----------- |
| 1          | 101       | Alice        | Math        |

If the key is `(student_id, course_id)`:

- `student_name` depends only on `student_id`
- `course_name` depends only on `course_id`

So that table should be split.

Better structure:

```text
students(student_id, student_name)
courses(course_id, course_name)
enrollments(student_id, course_id)
```

### 2NF Key Concept: No Partial Dependency

This is the key 2NF rule:

- no partial dependency

> **What does "depend on" mean?**
> "Depends on" means "is determined by" — if you know X, you automatically know Y without needing anything else.
> Test: can two rows share the same X but have a different Y? If no, then Y depends on X.
> Example: knowing `course_id` alone tells you `course_name`. You don't need `student_id` too.

Meaning:

- non-key attributes must depend on the whole primary key
- a non-key attribute cannot depend on just part of a composite key

Quick check:

- if the primary key is `(A, B)`, each non-key column should need both `A` and `B`

### 3NF in Plain Language

Third Normal Form means:

- the table is already in 2NF
- non-key columns should depend on the key directly, not through another non-key column

Example:

| student_id | state_code | state_name |
| ---------- | ---------- | ---------- |
| 1          | CA         | California |

Here:

- `student_id` determines `state_code`
- `state_code` determines `state_name`

So `state_name` depends on `student_id` through `state_code`.
That is a transitive dependency.

Better structure:

```text
students(student_id, state_code)
states(state_code, state_name)
```

### 3NF Key Concept: No Transitive Dependency

This is the key 3NF rule:

- no transitive dependency

> **What does "depend on" mean here?**
> "Depends on" means "can be looked up from" — knowing X alone is enough to tell you Y.
> Example: knowing `state_code = CA` always gives you `state_name = California`. No other column is needed.
> So `state_name` depends on `state_code`, not on `student_id`. That's the transitive chain 3NF wants you to break.

Meaning:

- non-key attributes cannot depend on other non-key attributes
- non-key attributes should depend only on the primary key

Quick check:

- avoid chains like `key -> non-key A -> non-key B`

### Fast Mental Model

Use these questions:

- 1NF: is each cell holding just one value?
- 2NF: does every non-key column depend on the whole key?
- 3NF: does every non-key column depend directly on the key?

### One End-to-End Example

Start:

```text
student_id | student_name | course_id | course_name | teacher_name
```

After organizing:

```text
students(student_id, student_name)
courses(course_id, course_name, teacher_name)
enrollments(student_id, course_id)
```

After removing transitive dependency if needed:

```text
courses(course_id, course_name, teacher_id)
teachers(teacher_id, teacher_name)
```

### Normalization Summary

| Normal Form | Simple question                                    |
| ----------- | -------------------------------------------------- |
| 1NF         | Is every cell atomic?                              |
| 2NF         | Does every non-key column depend on the whole key? |
| 3NF         | Does every non-key column depend only on the key?  |

### Final Takeaway

Normalization is really about one idea:

- keep each fact in the right place

When each fact is stored in the right table and linked with keys, the database becomes easier to understand and safer to update.

## 4. Real-World Example: This Project's Schema

The `schema.prisma` file in this project is a real database. Let's use it to make everything concrete.

### What Tables Exist

Here are the six models from `prisma/schema.prisma`:

```prisma
model User {
  id    String  @id @default(cuid())
  name  String?
  email String? @unique
  posts    Post[]
  comments Comment[]
  hearts   Heart[]
}

model Post {
  id        String   @id @default(cuid())
  title     String
  userId    String
  user      User     @relation(fields: [userId], references: [id])
  comments  Comment[]
  hearts    Heart[]
}

model Comment {
  id     String @id @default(cuid())
  title  String
  postId String
  userId String
  post   Post   @relation(fields: [postId], references: [id], onDelete: Cascade)
  user   User   @relation(fields: [userId], references: [id])
}

model Heart {
  id     String @id @default(cuid())
  postId String
  userId String
  post   Post   @relation(fields: [postId], references: [id], onDelete: Cascade)
  user   User   @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([postId, userId])
}
```

Each model has an `id` field as its **primary key** (`@id`).

### Is It in 3NF? Yes.

Every non-key column in every table depends directly on that table's own `id` — and nothing else.

Examples:

- `Post.title` depends on `post.id` — not on the user, not on any other column
- `Comment.title` depends on `comment.id`
- `Heart` has a composite key `(postId, userId)` — the heart belongs to that exact pair, and there are no other non-key columns that depend on only one half

No transitive chains exist. The schema is clean 3NF.

---

### Reverse Engineering: What If It Were Broken?

#### 1NF Violation — storing multiple values in one cell

Imagine we didn't have a separate `Comment` table and instead stuffed all comments into the Post row:

```text
post_id | post_title | all_comments
p1      | My Post    | "Great post!, Nice one, Thanks"
```

`all_comments` holds three values in one cell. That violates 1NF.

**Fix:** give each comment its own row → the separate `Comment` table.

#### 2NF Violation — a column depending on only part of a composite key

The `Heart` table has a composite key: `(postId, userId)`.

Imagine we added `post_title` directly to the Heart table:

```text
postId | userId | post_title    ← VIOLATION
h1     | u1     | "My Post"
h1     | u2     | "My Post"
```

`post_title` only depends on `postId` — it ignores `userId` entirely. That is a **partial dependency**, which violates 2NF.

Notice the problem: if you rename the post, you now have to update every heart row that references it. That is exactly the bug normalization prevents.

**Fix:** remove `post_title` from Heart. Look it up from the `Post` table when needed.

#### 3NF Violation — a column depending on another non-key column

Imagine we added geographic data to User:

```text
user_id | email         | country_code | country_name   ← VIOLATION
u1      | alice@x.com   | US           | United States
u2      | bob@y.com     | US           | United States
```

`country_name` does not depend on `user_id`. It depends on `country_code`. That is a **transitive dependency**: `user_id → country_code → country_name`.

**Fix:** split it out:

```text
users(user_id, email, country_code)
countries(country_code, country_name)
```

### Foreign Keys: Why They Exist When You Split Tables

When you split one big table into smaller tables, you lose the connection between rows. A **foreign key** is how you re-attach them.

**Analogy:** imagine a library.

- Books are in one room.
- Borrowers are in another room.
- A slip of paper on the shelf says "this book is borrowed by card number 42."

The card number on that slip is the foreign key. It does not copy the borrower's name, address, or anything else — it just holds a reference number you can use to look them up.

---

#### What a foreign key actually is (neutral SQL)

At its core, a foreign key is just **a plain column that holds another table's id**.

```sql
-- neutral SQL — no framework, no ORM
CREATE TABLE posts (
  id      TEXT PRIMARY KEY,
  title   TEXT,
  user_id TEXT REFERENCES users(id)   -- ← this is the foreign key
);
```

`user_id` is an ordinary column. Its value is an `id` that exists in the `users` table. That's the whole idea — no magic.

---

#### How Prisma writes the same thing

Prisma splits the foreign key concept across **two lines**:

```prisma
model Post {
  userId String                                               -- line 1: the actual column (the FK value)
  user   User @relation(fields: [userId], references: [id])  -- line 2: tells Prisma HOW to join
}
```

- **`userId String`** — this IS the foreign key. It is a real column in the database that stores a value like `"clx1abc..."`.
- **`user User @relation(...)`** — this is NOT a column. It is Prisma's instruction: "when I say `post.user`, look up the `User` row where `User.id = post.userId`". It does not exist in the actual database table.

So the `@relation` line is a convenience layer on top. The foreign key itself is just `userId String`.

---

#### Side by side

```text
Neutral SQL                          Prisma equivalent
─────────────────────────────────    ──────────────────────────────────────────────────
user_id TEXT REFERENCES users(id)    userId String
                                     user   User @relation(fields: [userId], references: [id])
```

Same thing — Prisma just makes the join explicit so your code can write `post.user.name` instead of writing a SQL JOIN manually.

---

#### `Comment` holds two foreign keys at once

```prisma
model Comment {
  postId String   -- FK to Post
  userId String   -- FK to User
  post   Post @relation(fields: [postId], references: [id], onDelete: Cascade)
  user   User @relation(fields: [userId], references: [id])
}
```

A `Comment` knows which post it belongs to (`postId`) and which user wrote it (`userId`). Those two plain string columns are the foreign keys. `onDelete: Cascade` means: if the post is deleted, all its comments are automatically deleted too.

---

**Why not just copy the data instead?**

If `Post` stored `user_name` directly:

```text
post_id | post_title | user_name
p1      | My Post    | Alice
```

And Alice changes her name — now you have to update every single post she ever wrote. With a foreign key you update `User` once and every post automatically reflects it.

**Foreign keys in one sentence:** a foreign key is a plain column that stores another row's `id` — a reference, not a copy.

---

### Summary: The Schema in One View

```text
User ──< Post       one user writes many posts
User ──< Comment    one user writes many comments
User ──< Heart      one user can heart many posts
Post ──< Comment    one post has many comments
Post ──< Heart      one post can receive many hearts

Foreign keys used:
  Post.userId     → User.id
  Comment.postId  → Post.id
  Comment.userId  → User.id
  Heart.postId    → Post.id
  Heart.userId    → User.id
```

Each arrow is a foreign key. No data is duplicated — just references.

## 5. Common Sticking Points Explained

These are the exact questions that trip people up when first learning normalization.

---

### What Does "Depends On" Actually Mean?

"Depends on" means **"is determined by"** — if you know X, you automatically know Y without needing anything else.

**The test:** can two rows share the same X but have a different Y?

- If **no** → Y depends on X (knowing X always gives you the same Y)
- If **yes** → Y does not depend on X (X alone is not enough)

| Question                                                          | Answer                         | Conclusion                                     |
| ----------------------------------------------------------------- | ------------------------------ | ---------------------------------------------- |
| Can two rows have `state_code = CA` but different `state_name`?   | No — CA is always California   | `state_name` depends on `state_code`           |
| Can two rows have `state_code = CA` but different `student_name`? | Yes — many students live in CA | `student_name` does NOT depend on `state_code` |

**Alternative phrasing:** replace "depends on" with "can be looked up from."
`state_name` can be looked up from `state_code` alone → it depends on it.

---

### Why Is state_name the Transitive One, Not state_code?

Given this table:

```
student_id | state_code | state_name
1          | CA         | California
```

The chain is:

```
student_id → state_code → state_name
```

- `state_code` directly depends on `student_id` — knowing the student tells you their state code
- `state_name` directly depends on `state_code` — knowing `CA` tells you `California`
- `state_name` does **not** directly depend on `student_id` — you cannot jump from "student 1" to "California" without passing through `CA` first

`state_code` is not the problem — it is the **bridge**. `state_name` is the problem because it hitchhiked into the students table via that bridge.

**The real-world test:** if every student in California was deleted from the database, would the fact "CA means California" stop being true? No — it is a free-standing fact that has nothing to do with students. Storing it in the students table is putting it in the wrong place.

---

### Does Column Order Affect Dependencies?

**No.** Rearranging columns on screen changes nothing.

Whether the table is written as:

```
student_id | state_code | state_name
```

or:

```
student_id | state_name | state_code
```

the logic is identical. `CA` still always means `California` regardless of which column appears first. Dependencies live in the **data relationship**, not the layout.

**The check questions are unchanged by order:**

- "If I know the student, do I know the state code?" → Yes
- "If I know the state code, do I know the state name?" → Yes
- "Can I get the state name from the student without going through the code?" → No

---

### What If state_code Wasn't in the Table?

This is the sharpest question: if the table only had two columns:

```
student_id | state_name
1          | California
```

Then `state_name` **directly** depends on `student_id`. No transitive dependency. No 3NF violation. Completely fine.

The violation only exists **because `state_code` is also in the table**. Its presence reveals that `state_name` doesn't actually need the student — it only needs the code. `state_code` exposes the shortcut.

So the precise 3NF test is not just "does this column depend on the key" — it is:

> **Is there another non-key column in the same table that already determines this value?**

| Table                                | Problem? | Why                                                                                             |
| ------------------------------------ | -------- | ----------------------------------------------------------------------------------------------- |
| `student_id, state_name`             | No       | `state_name` depends directly on the key                                                        |
| `student_id, state_code, state_name` | Yes      | `state_code` (a non-key column) determines `state_name` — the pair reveals the transitive chain |

---

### Which Column Becomes the Key When You Split?

When you spot a transitive chain and split it into a new table, you have to choose which of the two coupled columns becomes the primary key of that new table. This choice is not always free.

---

#### Example 1 — both columns are unique (free choice)

```
students(student_id, state_code, state_name)
```

Transitive chain: `student_id → state_code → state_name`

Both `state_code` and `state_name` are unique — there is only one `CA`, and there is only one `California`. Either can be the primary key of the split-out table:

```
Option A:  students(student_id, state_code)  +  states(state_code, state_name)
Option B:  students(student_id, state_name)  +  states(state_name, state_code)
```

Both are valid 3NF. The choice is a design preference (shorter codes are usually preferred as keys for storage and indexing reasons, but normalization does not force it).

---

#### Example 2 — only one column is unique (choice is forced)

A Polish university table (from a real lecture slide):

```
przedmiot (subject) | pesel (national ID) | Nazwisko (surname)
Analiza             | 78071103350         | Kowal
Algebra             | 78071103350         | Kowal
Bazy Danych         | 68121103312         | Nowak
```

Transitive chain: `przedmiot → pesel → Nazwisko`

`pesel` is a Polish national ID — guaranteed unique per person, like a social security number.
`Nazwisko` is a surname — "Kowal" is as common as "Smith". Multiple lecturers can share it.

Because `Nazwisko` is not unique, it **cannot** be a primary key. The choice is forced:

```
✅ Wykładowcy(pesel, Nazwisko)   — pesel is unique, safe as PK
❌ Wykładowcy(Nazwisko, pesel)   — Nazwisko is not unique, two Kowals would collide
```

---

#### The rule

| Situation                       | Flexibility                                            |
| ------------------------------- | ------------------------------------------------------ |
| Both coupled columns are unique | Either can be the key — design choice                  |
| Only one is unique              | That one must be the key — no choice                   |
| Neither is unique               | You need a surrogate key (e.g. an auto-generated `id`) |
