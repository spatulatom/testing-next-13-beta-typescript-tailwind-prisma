## Why this guide exists

Prisma can look like a normal npm package, but it is not just plain JavaScript at runtime. It combines generated client code, dynamic loading, and platform-specific artifacts. That is the core reason bundler behavior matters.

This guide explains exactly:

- What happens with Webpack vs Turbopack
- What happens locally vs on Vercel
- Which issues are real Prisma/bundling issues vs unrelated warnings
- What config and coding patterns we standardize on in this repo

---

## TL;DR

- Prisma should run on the Node.js server runtime, not in client bundles.
- Bundling server code is fine, but Prisma packages often need to stay externalized.
- In Next.js 16, Prisma packages are already on Next’s auto externalization list, but we keep explicit config for clarity.
- Vercel does not bundle at request time. Bundling happens during build.
- Runtime warnings from `pg` SSL modes are database connection-string issues, not Turbopack bundling failures.

---

## The mental model: build-time vs runtime

### Build-time (local `next build` or Vercel build)

Next compiles server and client code into deployable output. This is where Webpack or Turbopack does module analysis and bundling decisions.

### Runtime (local `next dev` requests, or Vercel serverless/node function execution)

The built server output is executed. If a dependency was externalized, Node resolves it from `node_modules` at runtime.

Important: Vercel is not re-bundling Prisma on each request. Any bundling decision already happened during build.

---

## Why Prisma is special

Prisma usage in app code often looks like normal imports, but runtime behavior is different:

- Prisma Client is generated from `prisma/schema.prisma`
- The runtime may rely on dynamic resolution and non-trivial package internals
- Adapter + database driver behavior introduces runtime-only concerns

Because of this, treating Prisma exactly like a pure ESM utility package can break in certain bundling paths.

---

## Webpack vs Turbopack (practical difference for Prisma)

### Webpack (historically)

- More mature compatibility surface for older server-bundling patterns
- Some projects “just worked” without explicit config
- Less obvious when a package should have been explicitly externalized

### Turbopack (current default in Next 16)

- Faster and stricter module graph handling
- More likely to surface boundary issues early (server/client/runtime assumptions)
- Better to be explicit about external packages and server-only imports

### What this means for us

Turbopack is not “wrong” with Prisma. It is exposing constraints that were easier to miss before. Our approach is to make server boundaries and externalization explicit.

## Project standards in this repo

### 1) Keep Prisma externalized for server execution clarity

In `next.config.js` we keep:

- `serverExternalPackages: ['@prisma/client', 'prisma']`

Notes:

- In Next.js 16, these packages are already in Next’s auto opt-out list.
- We still keep explicit config because it documents intent and avoids ambiguity during upgrades.

### 2) Keep Prisma usage server-only

Allowed:

- Route Handlers
- Server Components
- Server Actions
- Shared server utilities

Avoid:

- Client Components
- Browser-only modules

### 3) Use type-only Prisma imports in shared/component files

If only types are needed, use `import type` so no runtime Prisma dependency can accidentally leak into bundles.

### 4) Ensure client generation in all environments

`package.json` keeps:

- `postinstall: prisma generate`

This protects local fresh installs, CI, and Vercel builds.

### 5) Remember: generation time vs runtime responsibilities

- `prisma` CLI is used to generate Prisma Client and types during install/build.
- `@prisma/client` is used at runtime to execute actual queries.

Important:

- Production does not regenerate Prisma types on each request.
- Production loads the already generated client and runs queries through that runtime client.

---

## Vercel deployment lifecycle with Prisma

1. Install dependencies (`npm install`)
2. Run `postinstall` -> `prisma generate`
3. Build app (`next build`) with Turbopack/Next build pipeline
4. Deploy built output
5. Handle requests in Node runtime

If Prisma is externalized, server runtime loads it via Node resolution from installed dependencies.

## Real-world issue triage (what failed?)

Use this matrix before assuming “Turbopack Prisma bug”.

### A) Runtime warning mentions SSL mode semantics (`pg`)

Likely cause:

- Connection string parameter choice, not bundling

Fix:

- Use `sslmode=verify-full` (or explicitly opt into libpq-compatible behavior if intended)

### B) Error: Prisma module not found / generated artifacts missing

Likely cause:

- `prisma generate` not run in environment
- Version mismatch among `prisma`, `@prisma/client`, `@prisma/adapter-pg`

Fix:

- Sync versions
- Run clean generate
- Ensure `postinstall` hook exists

### C) Error appears only when file becomes client-side

Likely cause:

- Prisma runtime import leaked into client graph

Fix:

- Move runtime Prisma code to server-only module
- Keep client/shared files on type-only imports

### D) Edge runtime route tries to use Node-only Prisma path

Likely cause:

- Runtime mismatch (Edge vs Node expectations)

Fix:

- Use Node runtime for Prisma-backed handlers/components, or use a supported edge strategy

## CRUD + Prisma types: when to use generated vs custom types

Use this as the mental model:

- Prisma schema is the source of truth for database shapes.
- Generated Prisma types are best when your input/output is still close to the DB model.
- Custom types are best when shaping data for UI/API boundaries, or when Prisma cannot infer the exact runtime shape.

### Create / Update / Delete (write side)

- Prefer generated input types from `Prisma.*CreateInput`, `Prisma.*UpdateInput`, and `Prisma.*WhereUniqueInput`.
- This keeps writes aligned with schema constraints and relation rules.

Use custom types only when:

- You need a request DTO that is intentionally different from DB inputs (for example, form payloads or public API contracts).
- You combine multiple operations into one higher-level action (for example, one request maps to several Prisma calls).

### Read (query side)

- For simple model reads, generated model types are usually enough.
- For reads with `include` / `select`, define a type alias with `Prisma.ModelGetPayload<...>`.

In this repo, examples include:

- `types/Post.ts` (`Prisma.PostGetPayload`)
- `types/UserPosts.ts` (`Prisma.UserGetPayload`)
- `app/[post]/singlepost.tsx` (`Prisma.PostGetPayload`)

These are custom aliases, but still Prisma-derived, so they stay synchronized with schema changes.

### When to create fully custom types

Create fully custom types when you need to:

- Hide sensitive/internal fields before sending to clients.
- Merge Prisma data with non-Prisma data (session, feature flags, external APIs).
- Represent aggregate/raw query results not directly modeled by Prisma payload helpers.
- Freeze a stable API contract that should not change every time the DB schema changes.

Rule of thumb:

- DB boundary (CRUD implementation): prefer generated Prisma types.
- App/UI/API boundary (transport/view models): prefer custom types, potentially mapped from Prisma results.

Practical bucket model:

- Read bucket (query output): use generated model types or `Prisma.ModelGetPayload<...>`.
- Write bucket (create/update/delete inputs): prefer `Prisma.*CreateInput`, `Prisma.*UpdateInput`, `Prisma.*WhereUniqueInput` for DB operations.
- Boundary bucket (forms, component props, API contracts): use custom types and map to Prisma inputs.

This avoids a common confusion:

- Prisma can be 100% type-safe for DB operations.
- Real apps still need custom types at boundaries where shapes differ from DB models.

## Two-layer type model (Prisma-direct vs boundary-custom)

Use this fast model when deciding what type to write.

### Layer 1: Prisma-direct layer (DB access)

Use Prisma-generated types directly in:

- Server Actions that call Prisma
- Server utilities/repositories that call Prisma
- Route handlers that call Prisma

Typical type choices:

- `Prisma.ModelGetPayload<...>` for read results with `include`/`select`
- `Prisma.*CreateInput`, `Prisma.*UpdateInput`, `Prisma.*WhereUniqueInput` for write inputs

Goal:

- Stay as close as possible to schema truth.

### Layer 2: Boundary-custom layer (app/UI/API)

Use custom types in:

- Component props
- Form DTOs/request payloads
- API response contracts
- View models (reshaped/filtered data)

Why custom types are needed here:

- Boundary shapes usually differ from raw DB shapes.
- You may hide fields, add derived fields, or merge non-Prisma data.

Goal:

- Keep public/UI contracts stable and intentional.

Quick rule:

- If code directly talks to Prisma, start with Prisma types.
- If code talks to UI/API/form boundaries, use custom types and map to/from Prisma types.

## Type-safety gotchas (TypeScript + Prisma generics)

Sometimes an invalid field in `prisma.model.create({ data: ... })` may or may not be reported depending on TypeScript version/editor setup.

Why this happens:

- Prisma method signatures are generic.
- TypeScript excess-property checks can vary across versions and inference paths.
- Switching from VS Code bundled TypeScript to workspace TypeScript can change diagnostics.

What to do for stable strictness:

- Prefer using workspace TypeScript version consistently.
- Pin TypeScript version in `package.json`.
- After TS version changes, restart TS server.
- Force exact checks on Prisma `data` objects when needed.

Example strict pattern:

```ts
import type { Prisma } from '@prisma/client';

const data = {
  title: sanitizedTitle,
  userId: prismaUser.id,
  postId,
} satisfies Prisma.CommentUncheckedCreateInput;

await prisma.comment.create({ data });
```

This pattern catches accidental extra fields earlier and more consistently.

## Auth + Prisma: who defines what?

- NextAuth/Auth.js Prisma adapter uses your Prisma models.
- It does not auto-write your Prisma schema for you at runtime.
- You define auth models in `schema.prisma`, then run migrations/generate.

In this repo:

- Auth storage models are in `User`, `Account`, `Session`, `VerificationToken`.
- Adapter wiring is in `auth.ts` via `PrismaAdapter(prisma)`.

Terminology note:

- Google OAuth login is not strictly "passwordless" in the pure sense.
- It is OAuth/social login (no app-local password, but user still authenticates with Google credentials).

## Quick checklist before opening a bug

- Are Prisma packages version-synced?
- Did `prisma generate` run in this environment?
- Is Prisma runtime code only in server paths?
- Are Prisma type imports type-only where appropriate?
- Is `serverExternalPackages` set (even if auto-covered by Next)?
- Is the observed message actually a DB/connection warning instead of a bundling error?
